## 这是什么?

KarvyLoop 是一个**跑在你自己机器上的 AI agent 运行时**。你组建一支团队 —— 业务**域**(像公司)里配**角色**(有身份和偏好的 agent),每个角色由**原子**(单一职责的子 agent)组成 —— 内置助手**小卡 🦫** 用大白话替你把它们编排起来。它替你跑重复的活、**自己验证产出**,并把**每一次使用都结晶成一个版本的_你_**(技能、知识、决策偏好);同时一条硬的**人在环**规则(AI 提议、*你*拍板)让你始终坐在驾驶座上。

别的 agent 框架大多是让*单次* LLM 调用更可靠;KarvyLoop 是 **loop 原生**:设计单元是整条自运转的循环 —— *发现工作 → 干 → 验证 → 复利 →(你拍板)→ 重复*。配套对话界面叫 **KarvyChat**。

> 多数 AI 工具把你架空、烧钱、还千人一面。KarvyLoop 把你留在驾驶座、用得起、把每一次都沉淀成抄不走的你。

## 特性

- 🤖 **在本地组一支 AI agent 团队。** 建**域**(公司)、配**角色**(agent)、由可复用**原子**(子 agent)组成 —— 一套操作系统式的 L0→L4 实体阶梯。
- 🧭 **大白话编排,你来批。** 跟小卡说*"去产品域找几个人分析下竞品"* —— 它拆出形状:单点委派、**圆桌**(几个角色并行思考再收敛)、**工作流**(多步管线)、或对系统本身的**运维**自检,并返回一张**决策卡**。不确认就不动手(**H2A**)。
- 🔁 **loop 原生设计。** 两条性质相反的 loop —— *执行 loop*(原子干活、自验;可全自动)和*决策 loop*(角色↔你;刻意**不**自动化)。架构里 role/atom 这一刀,切的就是这两条。
- 🧠 **一切复利成"你"。** 过了验证门的运行结晶成可复用**技能**;你的接受/拒绝/修改结晶成**决策偏好**,预对齐未来的提案。这就是护城河 —— 代码可抄,**实例抄不走**。
- 📚 **个人知识库 + 认知图谱。** 喂它链接或笔记,它蒸馏成可检索的**知识点**并落到网状图谱(grep + 概念重叠,**无向量库**),控制台里能浏览。
- 🔌 **多 provider LLM 网关 + MCP。** 任意 Anthropic / OpenAI 兼容端点;唯一网关咽喉计每一个 token。接入 **MCP** server,其工具即达每个 agent。
- 🔒 **安全是构造级的。** 每个任务带能力令牌;文件/网络/进程访问对照它放行;第三方技能在 **bubblewrap**(Linux)/ **Seatbelt**(macOS)沙箱里跑,且带**完整性锁**(被篡改的技能目录在索引和执行两道都被拒)。控制台拒绝跨源浏览器请求(HTTP + WebSocket 同源门),叠加跨设备访问令牌。技能面板的「**能力总览**」卡一张表审计全能力面。它低于 agent 的信任边界 —— 绕不过。
- 🏠 **本地优先、私密。** 跑在 `localhost`;你的数据在 `~/.karvyloop/`,绝不上传。**MIT** 许可;**按版本发布**,没你点一下绝不自升。

### 支持的平台

| 系统 | 状态 |
|------|------|
| **Linux** | ✅ 一等公民 —— 完整安全沙箱(bubblewrap)。 |
| **macOS** | ✅ 支持 —— 系统自带 Seatbelt 沙箱(`sandbox-exec`),与 Linux 同套 fail-closed 契约;比 Linux 新、更糙。 |

## 快速开始

**环境**:Python 3.11+。**安装命令各平台一样**(`pip install -e .`);只有沙箱隔离原语不同 —— **Linux 需装 `bubblewrap`**(如 `apt install bubblewrap`),**macOS 用系统自带的 `sandbox-exec`,无需安装**;**Windows 暂无沙箱,以降级模式运行**(第三方技能脚本禁用,其余全功能)。其余全跨平台。

```bash
# 1) 安装 —— 把 `karvyloop` 装上 PATH、隔离(在 PEP 668「externally managed」发行版上也安全)
curl -fsSL https://raw.githubusercontent.com/Caprista/KarvyLoop/main/scripts/install.sh | bash
#    Windows(PowerShell):  irm https://raw.githubusercontent.com/Caprista/KarvyLoop/main/scripts/install.ps1 | iex
#    (要对着 clone 开发?  pip install -e .  —— 但见下方「karvyloop 命令找不到?」)
#    Linux 还需沙箱:  sudo apt install bubblewrap
#    macOS 沙箱自带:  无需安装(sandbox-exec 系统自带)

# 2) 配置模型(密钥放在仓库之外)
mkdir -p ~/.karvyloop
$EDITOR ~/.karvyloop/config.yaml   # 见下方"最小配置"

# 3) 启动本地控制台(网页 UI)
karvyloop console --host 127.0.0.1 --port 8766
# 浏览器打开 http://127.0.0.1:8766
```

**最小 `~/.karvyloop/config.yaml`**(把 `${ANTHROPIC_KEY}` 换成环境变量或字面量密钥——**真实密钥永不入库**):

```yaml
lang: zh
models:
  providers:
    anthropic:
      base_url: https://api.anthropic.com
      auth_header: x-api-key
      messages_path: /v1/messages
      api_key: ${ANTHROPIC_KEY}
      models:
        - id: anthropic/claude
          name: Claude
          api: anthropic-messages
          context_window: 200000
          max_tokens: 8192
agents:
  defaults:
    model: anthropic/claude
embedding:
  model: anthropic/claude
```

> 模型也能在控制台里图形化管理(左导航 🤖 模型)。任何 Anthropic 兼容端点都能接;OpenAI 兼容端点用 `api: openai-completions`。
> 只想看 UI 不接模型?`karvyloop console --no-llm`(只读,不需密钥)。

可组合:`pip install -e ".[mcp,web]"`。这些都不卡核心 loop——聊天、业务域/角色、决策、技能、token 账本在基础安装上全都能用。

## 头 15 分钟

不用懂 agent —— 整条 loop 一次走通:

1. **启动 + 接模型(约 5 分钟)。** `karvyloop console` → 配置页问你"AI 从哪来";选一个 provider,它给你"去拿 key(30 秒)"的链接,粘进去,它当场验证能不能用。(想本地跑?选本地选项,按安装指引来。)你的 key 在 `~/.karvyloop/config.yaml`,绝不进任何仓库。
2. **和小卡说话。** 在私聊里让它做点小而具体的事,它跑完返回 —— 这是**执行 loop**。
3. **组个团队。** 左导航 → 建一个业务**域**(像个公司),给它几个**角色**(比如域"数据组",角色"分析师"、"审核员")。
4. **把活交出去。** 回到和小卡的私聊,说一句*"把月度报表交给分析师做"*。小卡不越进域里——它**提议**转交,一张**决策卡**出现在 🤝。
5. **拍板。** 卡用你的语言说清"这是啥、凭什么";已核验区(✓/✗)和小卡的复述分开;你自己的标准预对齐在旁边。接受 / 改一条依据 / 拒绝 —— 你定。然后它进 🗳 **最近拍板**,供你回看。
6. **它会复利。** 你的接受/拒绝/修改结晶成**决策偏好**,预对齐未来的提案(你被问得更少、重复解释自己更少);重复的活结晶成**技能**,"快脑"直接复用——每次更便宜、更"你"。

## 理念 —— 为什么叫 "Loop 原生"

> 这一节讲*为什么*。如果你只想把它跑起来,上面已经够了——等你好奇它到底有什么不同,再回来看。

行业一路爬过这些范式:**prompt engineering → context engineering → harness engineering**。每一级都是让*单次* LLM 调用更可靠。今天的 agent 运行时都是 *harness 原生*——给一次调用套上脚手架(工具、ReAct、重试、护栏),让单个 agent 跑得稳。

KarvyLoop 是 **loop 原生**。设计单元不是"一次调用"也不是"一个 agent",而是整条自运转、会自查、会复利的循环:

```
发现工作 → 干 → 独立验证 → 复利(结晶) →(你拍板)→ 重复
      ↺ 然后回到开头 —— 每转一圈,loop 越来越便宜、越来越"你"
```

这听起来像是渐进的一小步。其实不是——因为关于 loop 有一个不太舒服的事实:

> **同一条 loop,在两个人手里,会跑出截然相反的结果。** 对一个人,它复利成杠杆;对另一个人,它一次一个"接受"地悄悄接管了判断,直到他再也分不清产出对不对——他在毫无察觉中被自动化掉了自己的工作。**loop 不是中性的。** 决定你落到哪一边的,是你究竟还当着这条 loop 的*工程师*,还是滑成了它的*旁观者*。

下面所有设计,都是为了让你**留在工程师那一侧**。

> **所有人都在造"越来越不需要你"的 agent;KarvyLoop 造的是"学你怎么拍板"的那一个 —— 并用"永不替你拍板"来证明。**

### 一条 loop,其实是两条

凑近看,一条运转的 loop 其实是**两条**性质相反的 loop:

- **执行 loop**——*怎么干 X。* 一个 atom(L1)只干一件事,能给自己的产出写验证门。这条 loop **可以完全自动化**,而且正在被商品化——大家的执行 loop 最终都长得差不多。
- **决策 loop**——*要不要干、按谁的标准干。* 它跑在某个 role(L2)和你之间。它**不能**被自动化,因为把它自动化掉,正是上面那种失败。你的意图、品味、责任,都活在这条 loop 里。

怎么分辨一件事属于哪条 loop?一把刀,从不含糊:**"这件事担不担你的责?"** 方向、取舍、认不认 → 决策 loop,留给你;纯粹把活干出来 → 执行 loop,交给机器。

所以 KarvyLoop **狠狠自动化执行 loop**(便宜、快、自验),却**拒绝自动化决策 loop**。架构里 role / atom 这一刀,切的就是这两条 loop。**H2A**(AI 提议、你拍板)不是一个能关掉的开关——它是"决策 loop 始终归你"的结构性保证。

认真对待决策 loop,还有一个推论:**如果你不得不主动去问"那件事怎么样了?",这条决策 loop 就已经塌了**——你沦为了系统的心跳检测器。这里的角色守一份**尽责下属**契约:遇到困难先自助穷尽(换方案、换原子、找技能、造技能);确实不可行才回来——**带着证据**回来("试了这三条路,卡在这里,需要你定这个"),绝不静默挂掉,绝不甩一句"失败了"完事。卡住的事自己冒出来;你的注意力只花在**决定**上,不花在**查岗**上。

### "你拍板"远不止一个是/否按钮

一个你无法*明智*行使的否决权,只是摆设。要真握住方向盘,你得**看懂**这个决策——但"看懂"不能等于"去翻工具日志"。系统做了什么、与你能从产出里看出什么,这中间的鸿沟就是 Don Norman 说的**评估鸿沟(Gulf of Evaluation)**;而最顺手的"补法"反而让它更糟:

- **过度信任陷阱**:一项关于 AI 解释的大型综述(微软,约 60 篇研究)发现,*更多*解释会增加依赖——**无论答案对不对**;一段流畅的说辞骗到了它本不配得到的信任。"那就多解释点"并不能让你掌控,反而把你哄睡。
- **过度判断 = 没判断**:问得太频繁,人要么终止 loop、要么全部橡皮图章——两者都是投降。在"问得太少(静默自动驾驶)"和"问得太多(决策疲劳)"之间,只有一条很窄的带。

KarvyLoop 的答案是一层**翻译层**,带两条硬规矩:

1. **接地,不诱导信任**:loop 声称"已解决"的,**只有**当它过了确定性验证门时,才能显示成已解决。其余一律标成小卡的*复述*,并显眼标注"未核验"。你的手机不是"因为存储满了所以卡"——而是告诉你**到底核验了什么、什么没核**。
2. **逼判断,不堆解释**:这个界面不试图让你*信任*它,而是逼你*判断*。它**很少**浮现(纯粹已核验的成功不打扰你);一旦浮现,它把你**自己以前的标准**摆在这次决策旁边,让你逐条 认 / 改 / 删,并且**不让高价值的"接受"被橡皮图章**。

落到具体,这就是**决策卡**:用你的语言说清"解决什么/怎么解";一个*已核验*区(✓ / ✗,可追溯到验证门)与小卡的复述**显眼分开**;你自己结晶出的标准**预对齐**在旁边;以及一道闸——拦住你正要随手挥过去的高价值"接受"。这些——而不是一个勾选框——才是"你始终是拍板的人"真正的构成。

> 已核验 ✓ 区只对**过了自动验证门**的步骤出现;很多步骤没有可自动核验的东西,就**老实标成"未核验"的复述,而不是假 ✓**。随着验证门接进越来越多的流程,这块会为你越来越多的工作亮起来——**我们宁可少给你绿,也不给你信不过的绿**。

### 会复利的楔子:两条 loop,两种结晶

两条 loop 不只是转——**各自会沉淀,而且沉淀的东西不同**:

- **执行 loop 结晶成技能**——*你怎么干活。* 每次运行都被观察;用够 + 过验证门后**结晶成持久技能**。下次"快脑"直接命中:更便宜、更贴合你。

  这里有一条我们视为承重墙的规矩:技能**存方法,绝不存答案**。把上季度查到的竞品清单缓存下来、半年后原样回放,那不是记忆——那是**给你自己的技能库投毒**(旧结论会过期)。所以技能里记的是*你是怎么做的*(从哪查、看什么、什么算够),每次命中都**拿当下的输入把方法重跑一遍**。复利的是你的做法——答案会过期,做法不会。

- **决策 loop 结晶成口味**——*你怎么拍板。* 你的接受 / 拒绝 / 修改结晶成标准,**预对齐**未来的提案,于是你被问得更少、重复解释自己更少。这些标准**刻意做得容易撤回**——一个相反的决策就削弱或退役它——因为目标是*贴合*你,绝不是把你*锁死*。

第一种,别人迟早也会做;第二种,几乎没人在做。它正是让这条 loop 变成**你的**、而不只是变得"好用"的原因。

### 底下那台飞轮:它到底怎么复利

结晶、"你始终拍板"——这些能成立,前提是底下有个东西**诚实地替你记分**。那个东西是一个基底——**Trace**——加上一刀有意的快慢分离:

- **唯一真相源**:一条 append-only 的 Trace 记下一切:每个角色/原子**做**了什么、你和系统**说**了什么、你和角色**决策**了什么、原子用什么**方法**做的、以及系统从这些里**学**到了什么(学习也回写进 Trace)。**所有评价都从 Trace 派生**,绝不在旁边另算。Trace 保持**可读**(让你、让安全/验证机制都能审计);护城河是"你的实例抄不走",绝不是"你自己也看不懂"。
- **快慢分离**:跑一个任务是急的,不该为"评价自己"买单——所以一次 drive **只执行 + 把事实写进 Trace**,一个耐心的学习器**离热路径**读 Trace 去评分。快的一侧从不等慢的;慢的那侧("越来越**像你**"的活)可以静下心慢慢来。(借的是大规模 RL 里 actor/learner 分离的**形状**——快 actor / 慢 learner、Trace 当日志,**不是它的机制:这里不训任何模型权重。**)
- **沿问责链评价**:角色对**你**负责,所以**你的决策评价角色**(这正是楔子里"决策偏好"那半);原子对它的**角色**负责——你甚至看不到它做了啥——所以**由角色用自己的客观指标评价原子**:子目标达成没?更快、更好、过验证门没?每一层都由它**对谁负责的那一层**来评,全部读自 Trace。

这就是"**记得你重复过什么**"(记忆)和"**学到什么对你真管用**"(护城河)的分界。被验证管用的方法开始**优先浮现**;每转一圈,这条 loop 就便宜一点、也更"你"一点。楔子是承诺——**这台飞轮是它兑现得了的证明**。

### 你的资产,不是别人的 agent

这里软件是消耗品——你描述要什么,它被写出来、用一下、丢掉。**留下**的是底下那层可复用的形状,结晶成归你的库:技能(*怎么做 X*)、决策偏好(*你怎么拍板*),以及你的活赖以构建的**角色与原子**。

已经在别处投入过 agent?带进来。导入的 agent 不会被压成一个文件——模型把它**拆解成一个角色**(它的人设)**加若干可复用原子**(它的能力),丢进一个共享池,你的任何角色都能组合。跨角色的近重复原子会被合并成一个规范原子——**你**确认合并,因为这是你的资产——每个原子还如实标注:*可执行*(工具真接上了)还是*顾问*(只靠人设推理),让"跑起来了"永远不会冒充"真能干"。

这个库——你的角色、原子、技能、决策风格——才是复利成杠杆、**抄不走**的那部分。

### 小卡跑团队,你定规矩

里面住着**小卡 🦫**,全局助手。用大白话跟它说——*"去产品研发域找几个人帮我分析下竞品"*——它会拆出形状:拉谁进来,这活是单点委派、**圆桌**(几个角色并行思考再收敛)、**工作流**(带交接的多步管线),还是对系统本身的**运维**自检。它从不直接*干*:每次编排都变成一张决策卡等你确认。执行扇出去自验,拍板留给你。

### 护城河:镜像 vs 实例

代码(**镜像**)开源、可抄——就是这个仓库。但你在它之上**长出来的实例**——你的记忆、技能、决策风格——归你、抄不走。开源与护城河自洽:我们公开镜像,绝不送出你的实例。

## 架构速览

**实体模型(操作系统式阶梯,与 `karvyloop/schemas/` 字段一一对应):**
- **L0 工具/技能** —— 无状态能力单元。一次性工具与结晶技能都在这层。
- **L1 原子** —— 最小"思考单元":能为它写验证门的单一职责 agent。这是**执行 loop**(可全自动)。
- **L2 角色** —— 原子 + 灵魂(身份/偏好)。角色↔人这条边界是**决策 loop**(不可自动化——价值复利之处)。
- **L3/L4 业务域/子域** —— 长期协作的"公司/部门",带共享价值观与护栏。

**运行时主干** —— 一个请求的流向:`控制台(FastAPI REST + WebSocket)→ MainLoop.drive → 快脑召回(命中=零 LLM)或 慢脑 Forge(ReAct)→ gateway(多 provider LLM)→ sandbox(bubblewrap)+ 能力令牌 → 流式回 UI`。gateway 是所有 LLM 调用的唯一咽喉,token 记账就在这一层做——记账绑在网关上(任何跟模型说话的路径都被计入),不是一个调用方可能忘掉的可选开关。

**安全是地基** —— 每个任务签发能力令牌;所有文件/网络/进程访问对照令牌放行;第三方技能脚本在沙箱里以最小授予运行(只限工作区,未经你显式授权无网络)。它低于 agent 的信任边界——绕不过。

完整细节请读源码——代码有注释,下面的地图告诉你去哪看。

### 决策审计流水

你的每次拍板(ACCEPT / REJECT / DEFER,以及显式撤回偏好 REVOKE)都记进一份**只追加、落盘**的流水(`~/.karvyloop/decision_log.json`,留存上限 5000 条)——因为在一个替你行动的系统里,**哪些是你拍的**必须答得出来。两个只读端点暴露它:

- `GET /api/decisions/recent?limit=N` —— 🗳「最近拍板」的 UI 回看窗(newest-first,≤50)。
- `GET /api/decisions/audit?since=<ts>&until=<ts>&decision=<ACCEPT|REJECT|DEFER|REVOKE>&limit=<N>` —— **给外部审计 / 合规用**:按时间窗 + 决定类型查完整留存流水。每条带 `ts, decision, summary, reason, kind, domain, role, proposal_id`;响应含 `total`(留存条数,看完整性)。把外部审计工具指向这个端点,或直接读那个 JSON 文件。

## 升级

KarvyLoop **按版本发布**([CHANGELOG](CHANGELOG.md)),有新版会**提示你**——但**绝不自己升级**。检测 → 提示 → *你*拍板。"绝不自己升级"是说系统不自作主张——不是说你得一直敲命令:你**一点**,它就把整套替你跑完。

- **怎么知道有新版**:控制台在有新版时显示一条**可关掉**的横幅;或随时跑 `karvyloop update`。(只是对 GitHub Releases 做一次纯版本查询——**零遥测、不发任何数据**;用 `KARVYLOOP_NO_UPDATE_CHECK=1` 可关掉。)
- **一键升级**:横幅上有个**升级**按钮——点一下,控制台自己把整条流程跑完(**停 → 装 → 起**)并重连页面,**不用敲命令**。(只限本机、防 CSRF、单实例;升不升仍是*你*定,只是省了命令。)
- **或者手动**:从 clone 装的 → `git pull && pip install -e .`;从 PyPI(将来发布后)→ `pip install -U karvyloop`。横幅也会按你的安装方式显示对的命令。
- **你的数据不丢**:你长出来的一切都在 `~/.karvyloop/`(config、beliefs、skills、角色/原子、决策流水)——在仓库之外——跨版本存活。破坏性的数据变更必随附迁移,并在发布说明里明确标出。
- **打包带走**:`karvyloop export` 把整个实例(技能、知识、偏好、历史)打成一个可携带压缩包——API key 刻意留在原机;新机器上解进 `~/.karvyloop` 就是回家。

## 前后端

经典分离:后端是 FastAPI(`/api/*` REST + 一条 `/ws` WebSocket,`/docs` 自带 OpenAPI);前端是 `karvyloop/console/static/` 下的纯静态 SPA,只通过 `fetch` + WebSocket 通信。换任意框架重写前端,照同一套契约做,后端零改动。

**构建(可选——只有改前端时才需要)。** console 原样服务 `karvyloop/console/static/`,且构建产物已入库,所以普通安装**不需要 Node**。SPA 正在**增量迁移到 TypeScript + Vite**(纯 TS,不上运行时框架)——源在 `karvyloop/console/frontend/`,每个迁好的模块构建成 `static/` 下的固定名 bundle,保持同一套 `window.Karvy*` 全局契约(未迁的 `.js` 照常用)。改了 TS 后重建:

```bash
cd karvyloop/console/frontend
npm install
npm run verify   # 类型检查 + 构建 → ../static + 运行时 smoke(jsdom)
```

> **访问与鉴权。** 默认 console 只绑 `localhost`(仅本机)。本机(loopback)请求免密;一旦你绑到局域网(`--host 0.0.0.0`)好从别的设备访问,**非本机地址必须带 token** —— token 每次启动新生成,在这台机器上跑 `karvyloop url` 获取带 token 的链接。(局域网**不是**信任边界 —— 否则公司/家里同网段任何人都能访问。)放到公网仍需你自己在前面加 TLS + 反向代理鉴权。
>
> **出门在外怎么访问?(远程访问。)** token 链接只在你的局域网内有效 —— KarvyLoop 刻意不开任何公网端口。本地优先阵营的标准做法是私有加密组网,如 [Tailscale](https://tailscale.com)(基于 WireGuard,个人免费):电脑和手机各装一次、登录同一账号,之后把 `karvyloop url` 链接里的 IP 换成主机的 Tailscale IP(仍需 `--host 0.0.0.0` + token),走到哪都能开。流量在**你自己的设备之间**端到端加密 —— 没有第三方服务器碰你的数据、不用配路由器端口转发、实例仍然只活在你自己的机器上。替代方案:纯 WireGuard(自建)或 Cloudflare Tunnel(需要域名)。**不建议**:在路由器上直接转发 8766 端口 —— console 的 token 鉴权不是为独自面对公网设计的。
>
> **`karvyloop` 命令找不到?** `pip install` 会把 `karvyloop` 命令装进你那个 Python 的 `bin`/`Scripts` 目录 —— 系统 Python 通常已在 `PATH` 上,但**若你装进了没激活的 venv、或 `pip install --user` 而 `~/.local/bin` 不在 `PATH`(不少发行版如此),就找不到**。两个办法:**(a)** 用**装它时那个 Python** 跑 `python -m karvyloop url`,不依赖 `PATH`,永远可用;**(b)** 想要干净的全局 `karvyloop` 命令,用 **pipx**:`pipx install karvyloop && pipx ensurepath`(pipx 隔离安装并把它加进 `PATH`)。pip 自己从不改你的 shell `PATH`。
