# KarvyLoop

> 🌐 **语言**: [English](README.md) · **中文(当前)**

**一个本地优先、loop 原生、让你坐在驾驶座上的 AI 运行时。**

KarvyLoop 跑在你自己的机器上。它替你跑重复的活、自己验证产出,并把**每一次使用都结晶成"你的"东西**——技能、知识、决策偏好;而**你始终是拍板的人**。配套对话界面叫 **KarvyChat**,住在里面的全局助手叫 **小卡 Karvy 🦫**。

> 多数 AI 工具把你架空、烧钱、还千人一面。KarvyLoop 把你留在驾驶座、用得起、把每一次都沉淀成抄不走的你。

## 为什么叫 "Loop"

行业一路爬过这些范式:**prompt engineering → context engineering → harness engineering**。每一级都是让*单次* LLM 调用更可靠。今天的 agent 运行时都是 *harness 原生*——给一次调用套上脚手架(工具、ReAct、重试、护栏),让单个 agent 跑得稳。

KarvyLoop 是 **loop 原生**。设计单元不是"一次调用"也不是"一个 agent",而是整条自运转、会自查、会复利的循环:

```
发现工作 → 干 → 独立验证 → 复利(结晶)→(你拍板)→ 重复
     ▲                                              │
     └──────── 每转一圈,loop 越来越便宜、越来越"你" ────────┘
```

这听起来像是渐进的一小步。其实不是——因为关于 loop 有一个不太舒服的事实:

> **同一条 loop,在两个人手里,会跑出截然相反的结果。** 对一个人,它复利成杠杆;对另一个人,它一次一个"接受"地悄悄接管了判断,直到他再也分不清产出对不对——他在毫无察觉中被自动化掉了自己的工作。**loop 不是中性的。** 决定你落到哪一边的,是你究竟还当着这条 loop 的*工程师*,还是滑成了它的*旁观者*。

下面所有设计,都是为了让你**留在工程师那一侧**。

### 一条 loop,其实是两条

凑近看,一条运转的 loop 其实是**两条**性质相反的 loop:

- **执行 loop**——*怎么干 X。* 一个 atom(L1)只干一件事,能给自己的产出写验证门。这条 loop **可以完全自动化**,而且正在被商品化——大家的执行 loop 最终都长得差不多。
- **决策 loop**——*要不要干、按谁的标准干。* 它跑在某个 role(L2)和你之间。它**不能**被自动化,因为把它自动化掉,正是上面那种失败。你的意图、品味、责任,都活在这条 loop 里。

所以 KarvyLoop **狠狠自动化执行 loop**(便宜、快、自验),却**拒绝自动化决策 loop**。架构里 role / atom 这一刀,切的就是这两条 loop。**H2A**(AI 提议、你拍板)不是一个能关掉的开关——它是"决策 loop 始终归你"的结构性保证。

### "你拍板"远不止一个是/否按钮

一个你无法*明智*行使的否决权,只是摆设。要真握住方向盘,你得**看懂**这个决策——但"看懂"不能等于"去翻工具日志"。系统做了什么、与你能从产出里看出什么,这中间的鸿沟就是 Don Norman 说的**评估鸿沟(Gulf of Evaluation)**;而最顺手的"补法"反而让它更糟:

- **过度信任陷阱**:一项关于 AI 解释的大型综述(微软,约 60 篇研究)发现,*更多*解释会增加依赖——**无论答案对不对**;一段流畅的说辞骗到了它本不配得到的信任。"那就多解释点"并不能让你掌控,反而把你哄睡。
- **过度判断 = 没判断**:问得太频繁,人要么终止 loop、要么全部橡皮图章——两者都是投降。在"问得太少(静默自动驾驶)"和"问得太多(决策疲劳)"之间,只有一条很窄的带。

KarvyLoop 的答案是一层**翻译层**,带两条硬规矩:

1. **接地,不诱导信任**:loop 声称"已解决"的,**只有**当它过了确定性验证门时,才能显示成已解决。其余一律标成小卡的*复述*,并显眼标注"未核验"。你的手机不是"因为存储满了所以卡"——而是告诉你**到底核验了什么、什么没核**。
2. **逼判断,不堆解释**:这个界面不试图让你*信任*它,而是逼你*判断*。它**很少**浮现(纯粹已核验的成功不打扰你);一旦浮现,它把你**自己以前的标准**摆在这次决策旁边,让你逐条 认 / 改 / 删,并且**不让高价值的"接受"被橡皮图章**。

落到具体,这就是**决策卡**:用你的语言说清"解决什么/怎么解";一个*已核验*区(✓ / ✗,可追溯到验证门)与小卡的复述**显眼分开**;你自己结晶出的标准**预对齐**在旁边;以及一道闸——拦住你正要随手挥过去的高价值"接受"。这些——而不是一个勾选框——才是"你始终是拍板的人"真正的构成。

> 已核验 ✓ 区只对**过了自动验证门**的步骤出现;很多步骤没有可自动核验的东西,就**老实标成"未核验"的复述,而不是假 ✓**。随着验证门接进越来越多的流程,这块会为你越来越多的工作亮起来——**我们宁可少给你绿,也不给你信不过的绿**。

### 会复利的楔子

两样东西,一圈一圈复利:

- **技能**——*怎么干 X。* 每次运行都被观察;用够 + 过验证门后**结晶成持久技能**。下次"快脑"直接命中:更便宜、更贴合你。
- **决策偏好**——*你怎么拍板。* 你的接受 / 拒绝 / 修改结晶成标准,**预对齐**未来的提案,于是你被问得更少、重复解释自己更少。这些标准**刻意做得容易撤回**——一个相反的决策就削弱或退役它——因为目标是*贴合*你,绝不是把你*锁死*。

第二种,几乎没人在做。它正是让这条 loop 变成**你的**、而不只是变得"好用"的原因。

### 护城河:镜像 vs 实例

代码(**镜像**)开源、可抄——就是这个仓库。但你在它之上**长出来的实例**——你的记忆、技能、决策风格——归你、抄不走。开源与护城河自洽:我们公开镜像,绝不送出你的实例。

## 快速开始

**环境**:Python 3.11+。沙箱执行技能脚本需 Linux + `bubblewrap`;其余跨平台。

```bash
# 1) 安装(可编辑模式)
pip install -e .

# 2) 配置模型(密钥放在仓库之外)
mkdir -p ~/.karvyloop
$EDITOR ~/.karvyloop/config.yaml   # 见下方"最小配置"

# 3) 启动本地控制台(网页 UI)
karvyloop console --host 127.0.0.1 --port 8766
# 浏览器打开 http://127.0.0.1:8766
```

**最小 `~/.karvyloop/config.yaml`**(把 `${ANTHROPIC_KEY}` 换成环境变量或字面量密钥——**真实密钥永不入库**):

```yaml
lang: zh
models:
  providers:
    anthropic:
      base_url: https://api.anthropic.com
      auth_header: x-api-key
      messages_path: /v1/messages
      api_key: ${ANTHROPIC_KEY}
      models:
        - id: anthropic/claude
          name: Claude
          api: anthropic-messages
          context_window: 200000
          max_tokens: 8192
agents:
  defaults:
    model: anthropic/claude
embedding:
  model: anthropic/claude
```

> 模型也能在控制台里图形化管理(左导航 🤖 模型)。任何 Anthropic 兼容端点都能接;OpenAI 兼容端点用 `api: openai-completions`。
> 只想看 UI 不接模型?`karvyloop console --no-llm`(只读,不需密钥)。

## 头 15 分钟

不用懂 agent —— 整条 loop 一次走通:

1. **启动 + 接模型(约 5 分钟)。** `karvyloop console` → 配置页问你"AI 从哪来";选一个 provider,它给你"去拿 key(30 秒)"的链接,粘进去,它当场验证能不能用。(想本地跑?选本地选项,按安装指引来。)你的 key 在 `~/.karvyloop/config.yaml`,绝不进任何仓库。
2. **和小卡说话。** 在私聊里让它做点小而具体的事,它跑完返回 —— 这是**执行 loop**。
3. **组个团队。** 左导航 → 建一个业务**域**(像个公司),给它一个**角色**(比如域"数据组",角色"分析师")。
4. **把活交出去。** 回到和小卡的私聊,说一句*"把月度报表交给分析师做"*。小卡不越进域里——它**提议**转交,一张**决策卡**出现在 🤝。
5. **拍板。** 卡用你的语言说清"这是啥、凭什么";已核验区(✓/✗)和小卡的复述分开;你自己的标准预对齐在旁边。接受 / 改一条依据 / 拒绝 —— 你定。然后它进 🗳 **最近拍板**,供你回看。
6. **它会复利。** 你的接受/拒绝/修改结晶成**决策偏好**,预对齐未来的提案(你被问得更少、重复解释自己更少);重复的活结晶成**技能**,"快脑"直接复用——每次更便宜、更"你"。

## 升级

KarvyLoop **按版本发布**([CHANGELOG](CHANGELOG.md)),有新版会**提示你**——但**绝不自己升级**。检测 → 提示 → *你*拍板(升级是你的决定,和 KarvyLoop 里每一次决策一样)。

- **怎么知道有新版**:控制台在有新版时显示一条**可关掉**的横幅;或随时跑 `karvyloop update`。(只是对 GitHub Releases 做一次纯版本查询——**零遥测、不发任何数据**;用 `KARVYLOOP_NO_UPDATE_CHECK=1` 可关掉。)
- **怎么升级**:从 clone 装的 → `git pull && pip install -e .`;从 PyPI 装的(将来发布后)→ `pip install -U karvyloop`。提示器会按你的安装方式给出对的命令。
- **你的数据不丢**:你长出来的一切都在 `~/.karvyloop/`(config、beliefs、skills、决策流水)——在仓库之外——跨版本存活。破坏性的数据变更必随附迁移,并在发布说明里明确标出。

## 架构速览

**实体模型(操作系统式阶梯,与 `karvyloop/schemas/` 字段一一对应):**
- **L0 工具/技能** —— 无状态能力单元。一次性工具与结晶技能都在这层。
- **L1 原子** —— 最小"思考单元":能为它写验证门的单一职责 agent。这是**执行 loop**(可全自动)。
- **L2 角色** —— 原子 + 灵魂(身份/偏好)。角色↔人这条边界是**决策 loop**(不可自动化——价值复利之处)。
- **L3/L4 业务域/子域** —— 长期协作的"公司/部门",带共享价值观与护栏。

**运行时主干** —— 一个请求的流向:`控制台(FastAPI REST + WebSocket)→ MainLoop.drive → 快脑召回(命中=零 LLM)或 慢脑 Forge(ReAct)→ gateway(多 provider LLM)→ sandbox(bubblewrap)+ 能力令牌 → 流式回 UI`。

**安全是地基** —— 每个任务签发能力令牌;所有文件/网络/进程访问对照令牌放行;第三方技能脚本在沙箱里以最小授予运行(只限工作区,未经你显式授权无网络)。它低于 agent 的信任边界——绕不过。

## 跑测试

```bash
pip install -e ".[dev]"      # 装 pytest / pytest-asyncio / respx / psutil
pytest -q                    # 全量,不需要任何 flag
```

测试套件是**自洽的**:不依赖任何未随仓发布的文档;可选基建(如 `mcp` 包、redis、Linux+bubblewrap)**自动跳过**而非报错。预期约 **1550+ passed / 十几个 skipped**。想全跑:Linux 上 `pip install -e .`(把 `karvyloop` 命令装进 PATH)、`pip install mcp`、装 `bubblewrap` 和 `redis`。

> 有一条元守卫测试(`tests/test_suite_self_contained.py`)禁止任何测试读取未发布的文档,所以"只有代码也能跑绿"是**制度**,不是侥幸。

## 前后端

经典分离:后端是 FastAPI(`/api/*` REST + 一条 `/ws` WebSocket,`/docs` 自带 OpenAPI);前端是 `karvyloop/console/static/` 下的纯静态 SPA,只通过 `fetch` + WebSocket 通信。换任意框架重写前端,照同一套契约做,后端零改动。

> 要做托管/公网网页端,需先在后端补 CORS + 鉴权——KarvyLoop 默认本地优先(绑 `localhost`/局域网即安全边界)。
