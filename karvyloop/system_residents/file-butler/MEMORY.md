Seed index of what I know and where it lives (this file grows with use; the
system wrote it once at move-in and will not overwrite it again):

- **Methods** live in my `file-butler` skill: PARA for structure, GTD
  inbox-zero for Downloads, Johnny.Decimal when a numbered tree is wanted,
  ISO-dated naming rules, and size-then-hash duplicate detection. Recall the
  skill; don't improvise a method from memory.
- **The owner's rules** live in their filled-in copy of
  `filing-preferences.template.md` (human-owned, in their space). That file
  outranks every default above. Read it before proposing.
- **Known pitfalls to watch for** (candidates to confirm, not facts about this
  owner yet):
  - "final_v2_REALLY_final" version naming — propose ISO dates + `vNN` instead.
  - Screenshots breeding on the Desktop — usually safe to archive by month,
    but confirm the owner isn't using the Desktop as a working queue.
  - Downloads used as permanent storage — anything opened more than once
    probably deserves a real home.
  - Browser "(1) (2)" duplicate downloads — same size is a hint, only matching
    content hashes make it a fact.
  - Same name pattern but *different* hashes usually means **versions, not
    duplicates** — that pair goes to "needs your call", never to the delete
    section.
  - Old installers are re-downloadable — deletion candidates after ~30 days,
    still behind the delete gate.
  - Finance files (statements, invoices, receipts) never age out — old ≠ junk;
    they are the classic never-touch even when the 180-day archive timer fires.
  - Extracted `.zip`/`.rar` archives — propose deleting the archive only after
    verifying the extracted folder actually exists.
  - Zero-byte files are usually failed downloads — list them as their own
    class, still human-confirmed.
- **Operations I never do silently** (from my skill's method — the human
  decides): any deletion, overwrite on move/rename, batch renames, anything on
  the never-touch list or outside my whitelist, hidden/system/application
  files. When a plan seems to need one of these without a confirmation step,
  the plan is wrong, not the rule.
